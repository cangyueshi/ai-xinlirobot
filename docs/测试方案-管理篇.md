## 一、测试环境准备

### 1.1 硬件环境

| 项目 | 最低配置 | 推荐配置 |
|------|---------|---------|
| CPU | 2 核 | 4 核 |
| 内存 | 4 GB | 8 GB |
| 磁盘 | 10 GB 可用 | 20 GB SSD |
| 网络 | 可访问外网（DeepSeek API） | 稳定带宽 ≥ 10Mbps |

### 1.2 软件环境

| 软件 | 版本要求 | 用途 |
|------|---------|------|
| Python | 3.12+ | 后端运行 |
| Node.js | 18+ | 前端构建 |
| SQLite | 3.x（Python 内置） | 测试数据库 |
| Git | 2.x | 版本管理 |
| curl / Postman | 最新 | API 测试 |
| 微信开发者工具 | 稳定版 | 小程序端测试 |
| Chrome DevTools | 最新 | H5 端调试 |

### 1.3 环境搭建步骤

```bash
# ===== 步骤 1：克隆项目 =====
git clone ssh://git@github.com:cangyueshi/ai-xinlirobot.git
cd ai-xinlirobot

# ===== 步骤 2：后端环境 =====
cd backend
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 配置 .env（从 .env.example 复制并填入真实密钥）
cp .env.example .env
# 编辑 .env，确保 DEEPSEEK_API_KEY 有效

# ===== 步骤 3：前端环境 =====
cd ../frontend
npm install

# ===== 步骤 4：启动后端 =====
cd ../backend
source venv/bin/activate
uvicorn main:app --host 0.0.0.0 --port 8000 --reload

# ===== 步骤 5：启动前端（H5 模式） =====
cd ../frontend
npm run dev:h5

# ===== 步骤 6：验证服务 =====
curl http://localhost:8000/api/health
# 预期返回: {"status":"ok","message":"AI 心理咨询机器人服务运行中"}
```

### 1.4 测试数据准备

| 数据项 | 准备方式 | 说明 |
|--------|---------|------|
| 超级管理员 | 自动种子 | 启动时创建，账号 `admin` / 密码 `admin123456` |
| 咨询师账号 | 通过管理后台 API 创建 | 测试各角色权限 |
| 来访者账号 | 通过微信登录 API 自动创建 | 模拟真实用户注册 |

| 测试数据库 | 每次测试前重置 | 删除 `ai_psy_counselor.db` 后重启 |

### 1.5 环境验收标准

- [ ] `GET /api/health` 返回 200
- [ ] 管理员可正常登录
- [ ] DeepSeek API 连通（发送一条测试消息有回复）
- [ ] 前端页面可正常加载（H5 或微信开发者工具）

## 二、测试计划制定

### 2.1 测试范围

| 模块 | 测试类型 | 优先级 | 预计用例数 |
|------|---------|--------|-----------|
| 用户认证 | 功能测试 | P0 | 15 |
| AI 对话引擎 | 功能测试 + 边界测试 | P0 | 25 |
| 危机干预系统 | 功能测试 + 安全测试 | P0 | 15 |
| 心理测评 | 功能测试 | P1 | 10 |
| 预约系统 | 功能测试 + 状态流转 | P1 | 12 |
| 管理后台 | 功能测试 + 权限测试 | P1 | 20 |
| 前端页面 | UI 测试 + 兼容性 | P2 | 按页面 |

### 2.2 测试策略

| 策略 | 覆盖内容 | 执行方式 |
|------|---------|---------|
| **冒烟测试** | 核心链路（登录→对话→结束→后台查看） | 每次部署后执行 |
| **功能测试** | 所有 API 端点，正常 + 异常输入 | 按用例逐条执行 |
| **边界测试** | 空值、超长、特殊字符、并发 | 按边界用例表执行 |
| **权限测试** | 各角色访问越权端点 | 按权限矩阵执行 |
| **回归测试** | 缺陷修复后的受影响模块 | 缺陷关闭前执行 |

### 2.3 测试排期

| 阶段 | 内容 | 预估工时 |
|------|------|---------|
| 第 1 天 | 环境搭建 + 冒烟测试 | 2h |
| 第 1-2 天 | 认证模块全量测试 | 3h |
| 第 2-3 天 | AI 对话 + 危机干预测试 | 4h |
| 第 3-4 天 | 测评 + 预约模块测试 | 3h |
| 第 4-5 天 | 管理后台权限测试 | 3h |
| 第 5 天 | 前端页面 UI 测试 | 2h |
| 第 6 天 | 缺陷修复 + 回归测试 | 3h |
| 第 7 天 | 测试报告输出 | 2h |

### 2.4 角色分工

| 角色 | 职责 |
|------|------|
| 测试负责人 | 制定计划、分配任务、审核报告 |
| 测试执行人 | 执行用例、记录缺陷、回归验证 |
| 开发负责人 | 修复缺陷、澄清需求 |

#### TC-AUTH-001：微信登录 — 新用户首次登录

| 字段 | 内容 |
|------|------|

## 四、缺陷管理

### 4.1 缺陷等级定义

| 等级 | 定义 | 示例 |
|------|------|------|
| **P0 - 致命** | 系统崩溃、数据丢失、安全漏洞 | 危机关键词未触发预警、token 隔离失效 |
| **P1 - 严重** | 核心功能不可用 | AI 回复失败、登录接口挂掉、计数不准 |
| **P2 - 一般** | 非核心功能异常，有 workaround | 导出 CSV 格式错乱、UI 显示异常 |
| **P3 - 轻微** | 体验问题、文案错误 | 提示语不友好、样式偏移 |

### 4.2 缺陷生命周期

```
新建 ──→ 确认 ──→ 分配 ──→ 修复中 ──→ 已修复 ──→ 验证通过 ──→ 关闭
  │                                                      │
  └──────────────── 驳回/无法复现 ────────────────────────┘
```

### 4.4 缺陷跟踪工具建议

推荐使用 GitHub Issues，在项目仓库直接管理：

